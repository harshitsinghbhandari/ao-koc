## Control Theory / PID for resource pressure

### The native setting first

Control theory is the engineering discipline of making systems behave the way you want despite disturbances. The canonical example is a thermostat: you want a room at 22°C, the outside temperature varies, and you have a heater you can turn up or down. The question is: *how much* do you turn the heater up, and *when*?

A naive answer is bang-bang control: if temperature < 22, heat at full power; if ≥ 22, stop. This works but produces oscillation — the room overshoots to 25°C, heater stops, drops to 19°C, heater blasts again. In systems with inertia (thermal mass, or in APEX's case, in-flight agent tasks), bang-bang control is unstable.

A **PID controller** solves this with three terms that are summed to produce the control signal:

**P — Proportional.** React to the current error. If context window is at 90% and your target is 70%, the error is 20%. The P term produces a control signal proportional to that: `u_P = Kp × error`. Large error → strong correction. Small error → gentle correction. Problem: proportional control alone never fully eliminates steady-state error — it backs off as error shrinks, so it reaches equilibrium slightly above target.

**I — Integral.** React to accumulated error over time. If context window has been above target for 10 consecutive steps, the integral term grows, producing increasing pressure to correct. This eliminates steady-state error — the integral keeps pushing until error is truly zero. Problem: if the system overshoots, the integral has accumulated a large value and keeps pushing in the same direction even as error reverses — this is *integral windup*, a real failure mode you need to design against.

**D — Derivative.** React to the *rate of change* of error. If context pressure is rising fast, the D term fires before it crosses the threshold — it's predictive. If pressure is falling fast, D term reduces correction to prevent overshoot. This is the dampening term.

The full controller: `u(t) = Kp·e(t) + Ki·∫e(t)dt + Kd·de/dt`

The art is tuning Kp, Ki, Kd. Too much P → oscillation. Too much I → windup and overshoot. Too much D → noise amplification (derivative of noisy signals is very noisy). Standard tuning methods (Ziegler-Nichols, Cohen-Coon) give starting points, then you tune empirically.

---

### Now the APEX mapping — three controllers, three pressure dimensions

APEX has three resource dimensions that need active control, not threshold monitoring. Each gets its own PID controller. They share a critical property: they all have *inertia* — you can't instantly shed context, instantly kill in-flight agents, or instantly free L5 sandbox capacity. Bang-bang control will make all three oscillate badly.

---

**Controller 1: Context window pressure (L2)**

The controlled variable is context window utilization — the fraction of the token budget currently occupied. The setpoint is your target utilization, say 72% (leaving headroom for the current agent's output). The actuator is L2's eviction and admission decisions.

```
error(t)     = current_utilization(t) - setpoint (e.g. 0.72)
u_P          = Kp × error(t)
u_I          = Ki × Σ error(t)          [accumulated pressure]
u_D          = Kd × (error(t) - error(t-1))  [rate of change]
eviction_pressure(t) = u_P + u_I + u_D
```

`eviction_pressure` is a continuous signal that L2 uses to score which artifact versions to evict. High pressure → evict aggressively, starting with low-liveness artifacts. Low pressure → admit freely. The D term is particularly valuable here: if a new batch of LSP graph data from L1 is incoming (L2 can see this on the event bus before it arrives), the derivative fires preemptively and L2 starts making room before the pressure spike hits.

**Integral windup guard:** Cap the integral term. If the integral has accumulated beyond a ceiling (meaning the system has been overloaded for too long), clamp it and alert L8. Uncapped integral windup causes L2 to evict everything including genuinely live artifacts — the classic overcorrection failure.

---

**Controller 2: Agent dispatch rate (L4)**

The controlled variable is the queue depth of pending tasks waiting for agent workers. The setpoint is a target queue depth — low enough that tasks aren't waiting long, high enough that workers are never idle. The actuator is L4's dispatch rate: how aggressively it pulls tasks from the DAG frontier and assigns them to agents.

The key insight here is that this controller operates on the *DAG execution frontier* — the set of tasks whose dependencies are all satisfied and which are ready to run. A growing frontier with a small worker pool means queue depth rises. A shrinking frontier (few ready tasks) means workers go idle.

```
error(t)      = queue_depth(t) - target_depth
dispatch_rate = base_rate - (Kp·e + Ki·Σe + Kd·Δe)
```

The D term here catches a subtle failure mode: if L6 starts returning failures rapidly, queue depth spikes because failed tasks re-enter the queue. The derivative fires immediately — L4 slows dispatch before the worker pool is swamped with retries, buying L3 time to replan.

---

**Controller 3: L5 sandbox resource utilization**

The controlled variable is aggregate sandbox resource utilization (CPU + memory across local/cloud workers). The setpoint is your target utilization ceiling — high enough to keep workers busy, low enough to prevent OOM kills and timeout cascades. The actuator is the rate at which L4 spawns new sandbox instances.

This controller has the longest time constant — sandbox provisioning has real latency, especially for cloud instances. The I term accumulates slowly and the D term needs a longer lookback window. Standard PID tuning for high-inertia systems: lower Kp, higher I time constant, aggressive D.

---

### The critical addition: feedforward

Pure PID is *reactive* — it responds to errors after they occur. For APEX you can do better because L3's DAG gives you *advance knowledge* of future load. If L3 has just planned a subtree with 8 parallel tasks, you know sandbox demand is about to spike — before it does.

**Feedforward control** adds a predictive term based on known future inputs:

```
u_total = u_PID + u_feedforward
u_feedforward = K_ff × predicted_load_from_DAG_frontier
```

L3 publishes its DAG frontier expansion events to the event bus. L4 and L5 controllers subscribe and add feedforward terms. This is the control theory formalization of "look ahead in the plan before spawning resources" — and it's far more stable than reactive-only control.

---

### Stability analysis — why this matters architecturally

This is the part most agent system builders skip and then debug painfully in production. Your system has a feedback loop:

```
L6 failures → L3 replanning → larger DAG frontier → 
more L4 dispatch → more L5 load → more L6 failures
```

This loop has *gain* (each failure can generate multiple retry tasks) and *delay* (replanning takes time, sandbox provisioning takes time). A loop with gain > 1 and delay will oscillate or diverge — this is not a metaphor, it's a mathematical guarantee from the Nyquist stability criterion.

The Nyquist criterion says: a feedback loop is stable if, at the frequency where the loop's phase shift reaches 180°, the loop gain is less than 1. For APEX, this means: measure how much a single L6 failure amplifies (gain) and how long the replan-dispatch-execute-validate cycle takes (sets the phase shift). If gain × some function of delay > 1, the system will oscillate.

The practical implication for architecture: **your L3 replanning must include a damping term**. When replanning after failures, L3 should not immediately expand to the maximum possible DAG frontier. It should expand conservatively, proportional to available worker capacity reported by L4/L5 controllers. This is the derivative term applied at the planning level — rate-limiting the response to failures.
